02 Reference Analysis

Viktor

getviktor.com ↗

Commercial Slack/Teams AI coworker with 3,000+ integrations. @mention → end-to-end execution — PDFs, spreadsheets, deployed web apps, CRM updates.

▶ What to steal
  • @mention as primary UI. No dashboards, no portals. Slack is the UI.
  • Real deliverables. Artifacts, not paragraphs. Files, deploys, emails.
  • Deep memory. Conversations compound — learns preferences, context, past decisions.
  • Recurring task ownership. Proposes automations and schedules recurring work.
  • Proactive work discovery. "It figures out what needs automating."
▶ What's missing
  • No visibility into agent reasoning or intermediate steps
  • No ability to steer mid-task — fire-and-forget
  • No open architecture — can't add your own tools or models
  • No self-hosted option — SOC2 ≠ data sovereignty

OpenClaw

GitHub ↗

Self-hosted AI gateway. Single Node.js process, 20+ channels, built-in cron, sub-agent spawning, skills, sandboxing.

▶ Primitives we build on
Primitive What it gives us
Gateway + SlackNative Bolt integration, mention gating, thread binding
sessions_spawnIsolated agent sessions with task prompts, announced results
Cron schedulerPersistent jobs, one-shot or recurring, isolated execution

Tool policiesFilter which tools each agent/subagent can use
Memory + searchMEMORY.md + semantic search across memory files

▶ Gaps to fill
  • No built-in task queue / work discovery engine
  • No structured task state (just session transcripts)
  • No integration framework beyond skills (no OAuth marketplace)
  • No approval workflow for agent-proposed actions
  • Memory is file-based, not queryable structured storage

Work Discovery Engine

Cron-driven scanner (every 15–30 min). Reads Slack channels, Linear issues, GitHub PRs, calendar, web for actionable work.

Output: Candidate tasks ranked by priority × urgency × confidence. High-confidence → auto-dispatch. Low-confidence → human approval.

Task Dispatcher

Selects agent type (coder, researcher, writer, ops), builds task prompt with context, calls sessions_spawn.

Tracks: Task state in SQLite. Sets up delivery channel (thread reply, DM, or channel post).

Question Router

When a sub-agent hits a blocker, surfaces the question in the right Slack thread. Pipes human answers back via sessions_send.

Escalation: DM if no thread response within configurable timeout.

Task Store

SQLite-backed structured persistence. Queryable, relational, zero-ops. Not transcripts.

Task { id, source, title, status,
  priority, agentType, sessionKey,
  assignedTo, blockedOn, artifacts[],
  createdAt, updatedAt, completedAt }

04 Architectural Decisions

▶ ADR-1: Build on OpenClaw vs. From Scratch
Recommended: OpenClaw
Build on OpenClaw From Scratch
Pros Slack, spawning, cron, skills, sandboxing — all done. Active OSS community. Full control. No inherited complexity. Clean abstractions.
Cons Tied to session model. File-based memory. Single-process. 6+ months to rebuild existing primitives.

The primitives are there. We extend, not rebuild.

▶ ADR-2: Orchestrator as Skill vs. Plugin vs. Fork
Recommended: Hybrid
Skill Plugin Fork
Pros Simplest. Uses existing tools. Easy to distribute. Access to Gateway internals. Can add new tools. Full control.
Cons Limited to existing tools. No Gateway events. Coupled to internals. Maintenance burden. Diverges from upstream.

Skill teaches the agent how to orchestrate. Plugin exposes new tools (task.create, task.list, integration.query). Best of both.

▶ ADR-3: Task Store Backend
Recommended: SQLite
SQLite JSON Files Postgres
Pros Queryable, relational, single-file, zero-ops Simple, fits file-based pattern Full RDBMS, scales
Cons Needs driver Not queryable, no indexes Running server, over-engineered for v1

Queryable, zero-ops, single file. Migrate to Postgres if/when multi-instance is needed.

▶ ADR-4: Work Discovery — Pull vs. Push vs. Hybrid
Recommended: Hybrid

Cron-based discovery runs every 15–30 min for proactive scanning. Webhooks for real-time events from Linear, GitHub, etc. Slack messages hit the agent directly via OpenClaw's channel integration.

Pull (cron) Push (webhooks) Hybrid
Pros Simple, predictable, easy to debug Real-time, low latency Best of both
Cons Latency up to cron interval Complex, reliability concerns More moving parts
▶ ADR-5: Question Surfacing
Recommended: Thread + DM Escalation

Thread reply by default (context is colocated, others can help). Escalation to DM if no response within configurable timeout. Optional #agent-questions channel as overflow/dashboard.

▶ ADR-6: Agent Autonomy Level
Default: Propose + Approve
Full Auto

Low-risk, well-understood tasks. Reports, research, status updates.

Propose + Approve ★

Medium-risk. Agent proposes, human approves via ✅ reaction or reply.

Human-Initiated

High-risk or new task types. Only acts on explicit @mentions.

07 Risk Matrix

Context Pollution

High likelihood · High impact

Orchestrator's context window fills with task metadata, degrading performance.

Mitigation: Isolated sessions. Task Store is out-of-band (SQLite). Orchestrator reads on demand.

Runaway Costs

Med likelihood · High impact

Proactive agents spawn too many sub-agents, burning tokens.

Mitigation: Token budgets per task. Max concurrent sub-agents. Approval gates for expensive ops.

Question Loops

Med likelihood · Med impact

Agent asks, human answers, agent misinterprets, asks again.

Mitigation: Structured question format. Max rounds per task. Escalate to human takeover after N rounds.

Slack API Rate Limits

Med likelihood · Med impact

Too many reads/writes hit Slack's tier limits.

Mitigation: Batch reads. Cache channel history. Respect tier limits. Use OpenClaw's built-in chunking.

Stale Context

Low likelihood · Med impact

Work discovery acts on outdated Slack messages.

Mitigation: Timestamp filtering. Dedup against Task Store. Check if work is already in progress.

09 Open Questions

Q1

Multi-tenant or single-tenant?

Viktor is multi-tenant SaaS. OpenClaw is single-tenant self-hosted.

Rec: Single-tenant first, multi-tenant later via managed hosting.

Q2

Model selection per agent type?

Cheap models for discovery/triage, expensive for coding/research?

Rec: Yes, configurable per agent type.

Q3

How much Slack history to scan?

Full history is expensive. Last 24h? 100 messages?

Rec: Configurable, default 24h or 200 messages (whichever is less).

Q4

Own Slack bot identity or share OpenClaw's?

Separate bot app vs. same bot with different persona.

Rec: Same bot, different persona via SOUL.md. Avoids Slack app overhead.

Q5

Artifact storage?

Local filesystem? S3? Slack file uploads?

Rec: Local filesystem + Slack upload for delivery. S3 optional for persistence.

TL;DR

Build a Slack-first agent orchestrator as an OpenClaw skill + plugin, powered by: a Task Store (SQLite) for structured state, a Work Discovery Engine that scans Slack/Linear/GitHub/web, a Question Router that surfaces blockers in Slack threads, and configurable autonomy levels from full-auto to human-initiated.

Viktor proved the UX. OpenClaw provides the runtime. We build the orchestration layer in between.
