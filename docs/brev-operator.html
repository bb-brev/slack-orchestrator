03 Market Research & Value Proposition

How Companies Actually Track & Execute on Goals

OKRs are the dominant framework for strategy execution at growth-stage and enterprise companies. But the framework itself isn't the problem — the operational overhead of running it is.

The Typical OKR Cycle (Quarterly)

  1. Goal Setting (Week 1): Leadership sets company OKRs. Teams draft aligned team OKRs. Rounds of review & alignment. → 5–10 hours per manager
  2. Check-ins (Ongoing): Bi-weekly or weekly updates on each key result. Manual data entry or spreadsheet pulls. → 30–60 min/person/week
  3. Reviews (Monthly/Quarterly): Business reviews, QBRs, team standups. Prep decks, gather data, present. → 4–8 hours per review per team lead
  4. Follow-through: Action items from meetings. Tracking commitments. Escalating blockers. → Often nobody owns this
  5. Retrospective: What worked, what didn't. 71% of OKR users hold retrospectives, 91% apply learnings to next cycle. → The cycle repeats

Inspired by OpenClaw's sessions_spawn model — each task gets an isolated agent with the right tools, context, and permissions. Multi-tenant: agents can only see their own org's data.

The brain. Runs the planning loop. Evaluates org state, decides what needs attention, dispatches work agents.

Model: Opus/Sonnet · Runs: always-on per org

Model: Sonnet · Runs: on-demand, spawned per task

🔎 Analyst

Deep-dives into data. Finds trends, correlations, blockers. Answers "why is this KPI declining?" with evidence.

Model: Sonnet/Opus · Runs: on-demand

📣 Coordinator

Handles human interactions in Slack. Follow-ups, approvals, question routing. Knows when to DM vs. thread vs. channel.

Model: Haiku/Sonnet · Runs: event-driven

⚡ Executor

Takes approved actions. Updates goals, reassigns owners, creates cadence events, modifies KPI targets.

Model: Haiku · Runs: on approval, short-lived

🔌 Integrator

Pulls data from external tools (HubSpot, Linear, Jira, Sheets). Reconciles external state with Brev's model.

Model: Haiku · Runs: scheduled + on-demand

09 Orchestration Loops

Loop 1: Proactive Planning (Cron — every 2–4 hours)

1

Scan org state. Read all goals, KPIs, check-ins, cadences. Identify: goals at risk, overdue check-ins, stale action items, approaching deadlines.

2

Prioritize. Rank by impact (company goal vs. team goal), urgency (days until deadline), and staleness (days since last update).

3

Plan actions. For each item: draft a check-in, compose a follow-up, prepare an escalation, or schedule a deep-dive.

4

Execute. Spawn agents for each action. Drafters produce artifacts. Coordinators deliver to Slack. Track results.

Loop 2: Reactive Handling (Event-driven)

→

Slack @mention: Parse intent. Route to CRUD tool, query engine, or spawned agent. Respond in-thread.

→

Approval reaction (✅): Execute the draft action. Update Brev. Confirm in thread.

→

Thread reply to Operator: Context for a pending action. Human provides info, Operator continues.

→

Integration webhook: External state changed (PR merged, deal closed, metric updated). Recalculate, notify if threshold crossed.

Loop 3: Cadence Lifecycle (Meeting-driven)

①

Pre-meeting (1h before): Drop agenda in Slack. Include: open action items, metrics since last time, goals needing discussion, suggested topics.

②

During meeting: Listen to transcript (existing Brev feature). Extract action items, decisions, commitments in real-time.

③

Post-meeting: Create follow-up tasks in Brev. Assign owners. Set deadlines. Send summary to Slack. Begin tracking.

④

Between meetings: Follow up on commitments. "Hey @jake, in Tuesday's standup you said the API migration would be done by Friday — how's it going?"

11 Risk Matrix

Trust Erosion — Operator Nags Instead of Helps

High likelihood·High impact

If follow-ups feel like spam, users mute the bot. Game over.

Mitigation: Every message must carry value: a drafted artifact, specific context, actionable options. Never a bare "reminder: your check-in is due." Per-user cooldown. Mute controls.

Data Integrity — Operator Makes Wrong Updates

Med likelihood·High impact

Operator misinterprets a Slack message and updates a KPI incorrectly, or reassigns a goal to the wrong person.

Mitigation: All writes require approval. Undo capability. Audit log with "Operator did X because Y." Confidence thresholds — low-confidence actions always require explicit approval.

Multi-Tenant Cost — Per-Org Agent Sessions Are Expensive

Med likelihood·High impact

Running planning loops every 2 hours for every org, plus on-demand agents, burns tokens fast.

Mitigation: Cheap models (Haiku) for scanning/triage. Expensive models only for drafting/analysis. Per-org token budgets. Tiered pricing (more active Operator = higher tier).

Scope Creep — "Can the Operator Also..."

High likelihood·Med impact

Every customer will want the Operator to do something different. HR tasks, IT tickets, expense reports.

Mitigation: Hard scope boundary: the Operator operates within Brev's domain (goals, cadences, execution). External integrations are data sources, not action targets. Phase 1 = Brev CRUD only.

Slack API Limits

Low likelihood·Med impact

High-activity orgs might hit Slack rate limits with frequent messages and reactions.

Mitigation: Message batching. Thread consolidation. Rate-aware queue. Daily message budget per org.

▶ Q1 Pricing Model — Per-seat, per-org, or usage-based?

Research: How competitors price

The OKR/performance management market prices per-seat-per-month: Lattice ($6–11), 15Five ($4–14), Betterworks ($8–15), Quantive ($10–18). AI-agent products like Viktor charge $20–50/user/mo. Pure usage-based pricing (per action, per token) creates unpredictable bills that enterprise buyers hate — every procurement team we'd sell to wants a predictable line item.

Recommendation: Per-seat add-on tier

Brev Pro + Operator: $22–28/user/mo (a $10–16 premium over Brev Pro at $12/user/mo). This prices the Operator below a fractional ops hire ($80K–150K/yr for coordination work at a 50-person company) while keeping the pricing simple and predictable. Usage caps per tier prevent runaway AI costs: e.g., 500 Operator actions/org/month on standard, 2000 on enterprise.

Why not pure usage-based: Usage pricing works for developer tools (Stripe, Twilio) where the buyer is technical and understands per-call economics. Brev sells to ops leaders and executives who need a budget-friendly line item. Per-seat is how every competitor prices, so procurement teams expect it.

Enterprise: Custom pricing for 500+ seats. Includes dedicated agent tuning, SSO/SCIM, custom integrations, SLA, and priority support.

▶ Q2 DMs vs. channels — where should the Operator talk?

Research: Communication patterns in ops tools

Slack's own data shows DMs have 2–3× higher engagement rates than channel posts. But DMs create silos — managers can't see if their reports are engaging. 15Five and Lattice both use email DMs for check-in prompts (private) but post summaries to shared views (transparent). The best ops tools split by intent: personal accountability in private, team visibility in public.

Recommendation: Intent-based routing with defaults

Message TypeDefault SurfaceRationale
Check-in draftsDMPersonal, may contain sensitive performance data. User reviews privately before it goes on record.
Follow-ups on commitmentsDM (1st touch) → Thread (2nd)First follow-up is private. If no response, escalate to the thread where the commitment was made — adds social accountability.
Weekly digestsChannel (#leadership, #team-x)Team/company-wide visibility. Everyone should see the same report.
Risk escalationsDM to decision-makerDecision-maker needs to act without public pressure. If no response in 24h, escalate to channel.
Meeting agendasChannel thread or CanvasAll attendees need access before the meeting.

All defaults configurable per org. Admins can set a "transparency level": private (all DMs), balanced (default above), or open (all in channels).

▶ Q3 How much autonomy on day one?

Research: Trust calibration in AI agents

Every successful AI assistant product (GitHub Copilot, Cursor, Notion AI) follows the same trust arc: start with suggestions, graduate to autonomous actions as the user builds confidence. The research on automation trust (Lee & See, 2004) shows that overtrust is as dangerous as undertrust — users who over-rely on AI stop verifying outputs. The key is calibrated trust: the user understands exactly what the AI will do autonomously vs. what requires approval.

Critically, 83% of companies report positive OKR impact — but only with consistent use. If the Operator requires too much approval overhead, adoption drops and we're back to "tool that nobody uses." The right balance: automate the tedious (data gathering, drafting), require approval for the consequential (submitting, reassigning).

Recommendation: Three-tier autonomy with progressive unlocking

Always Autonomous (Day 1)
  • • Read all Brev data (goals, KPIs, cadences)
  • • Pull data from integrations
  • • Draft check-ins and reports
  • • Generate meeting agendas

▶ Q4 Should the Operator's personality be customizable per org?

Research: Tone in workplace AI

Microsoft's research on Copilot adoption found that tone mismatch is one of the top reasons users disengage with AI assistants. A startup with a casual Slack culture will reject corporate-speak. An enterprise with compliance requirements needs formal, precise language. Intercom and Drift both offer per-workspace tone customization for their AI chatbots — it's table stakes for multi-tenant B2B AI products.

However, personality customization is a v2 feature. The v1 risk is getting the default personality wrong and losing users before they try customization. The default must be: professional but warm, direct but not abrupt, helpful but not obsequious.

Recommendation: Ship one great default, add customization in v2

v1 personality: Confident, direct, context-rich. Modeled on the best Chief of Staff you've worked with — the one who shows up with the data already pulled, the options already framed, and the follow-up already drafted. Not chatty. Not robotic. Professional with a hint of warmth.

v2 customization axes:

  • • Formality: casual ↔ formal (affects word choice, emoji use, message length)
  • • Assertiveness: gentle ↔ direct (affects follow-up intensity, escalation speed)
  • • Verbosity: concise ↔ detailed (affects report depth, context inclusion)
  • • Proactivity: conservative ↔ aggressive (affects how often the Operator initiates)

Implemented via system prompt tuning per org — stored in the org's Operator config. Admin configures via Slack modal (/brev settings) or Brev web dashboard.

▶ Q5 Should the Operator surface in the Brev web app too?

Research: Multi-surface AI in B2B

Every B2B AI product that started in one surface eventually expanded. GitHub Copilot went from VS Code to CLI to web. Notion AI lives in both the editor and sidebar. Intercom's AI started in chat widgets, now appears in inbox, help center, and reporting. The pattern: the AI does its work in the user's flow (Slack), but the record lives in the system of record (Brev web app).

Crucially, admins and executives need oversight — "what has the Operator been doing?" This can't live only in Slack threads. An audit trail in the web app is a compliance requirement for enterprise customers.

Recommendation: Yes — Slack is interaction, Brev web is record

Brev web app additions:

  • • Operator Activity Feed: Timeline of all Operator actions — what it did, why, what triggered it, outcome. Filterable by team, user, action type.
  • • Operator Dashboard: Aggregate metrics — tasks completed, check-ins submitted, follow-ups resolved, approval rate, time saved.
  • • Operator Settings: Admin panel for org-wide Operator configuration — autonomy levels, channel routing, personality, active hours, mute rules.
  • • Deep Links: Every Operator message in Slack includes a "View in Brev" link that opens the relevant context in the web app (goal detail, check-in, report).

Not in v1: A full chat-with-Operator experience in the web app. Slack is the primary interaction surface. The web app is for oversight, configuration, and record-keeping. This avoids building two chat UIs and keeps the team focused.

▶ Q6 What happens when someone ignores the Operator?

Research: Escalation design in ops tools

PagerDuty's escalation research shows the optimal pattern: 3 touches max before escalation. Each touch adds urgency and context. After the 3rd touch, escalate to the next person in the chain — never continue nagging the same person. Jira's SLA engine follows a similar pattern: warn → breach → escalate. The worst possible behavior is infinite reminders — it teaches users to ignore all messages from the tool.

Crucially, non-response is itself a signal. If someone consistently ignores check-in prompts, the Operator should learn this pattern and suggest: (1) auto-completing from available data, (2) reassigning ownership, or (3) flagging to a manager — not sending a 5th reminder.

Recommendation: 3-touch escalation with smart adaptation

1

Touch 1: DM with drafted artifact (Day 0)

"Hey @mike, here's your Q2 pipeline check-in draft based on HubSpot data. Approve, edit, or dismiss." Includes pre-filled data so the user just has to click a button.

2

Touch 2: Thread nudge with added context (Day 2)

"Quick bump — this check-in covers a goal that's been At Risk for 2 weeks. Your team's QBR is Thursday and this data would be helpful for the review." Adds urgency and social context.

3

Touch 3: Escalate + flag (Day 5)

Operator notifies the manager: "@sarah, @mike's Q2 pipeline check-in has been pending for 5 days. Options: (1) I submit what I have from HubSpot data, (2) reassign to another team member, (3) skip this cycle." Simultaneously flagged in the weekly report as "unresolved."

∅

After Touch 3: Stop. Learn. Report.

No more messages about this item. Log the non-response. Include in weekly/quarterly analysis: "@mike responded to 2 of 8 check-in prompts this quarter." This data becomes the basis for org-level insights on engagement and process fit.

Timing, escalation chain, and max touches are all configurable per org. Default: 3 touches over 5 days.
