Infrastructure & Multi-Tenant Architecture

Architecture
Deep Dive

How the Brev Operator isolates tenants, executes agents, manages Slack integrations, and scales from 10 to 1,000 organizations — all on serverless AWS infrastructure orchestrated by SST.

SST v3 AWS Lambda Postgres RLS Multi-Tenant SQS / SNS

Part I

Multi-Tenant Design

01 Per-Org Isolation Model

Every Slack workspace that installs Brev Operator becomes a tenant (called an org). The isolation boundary is the org_id — a UUID generated at install time. Three things are scoped to it:

🔐 Agent Sessions

Every agent invocation carries org_id in its execution context. Agents can only read/write data belonging to their org. The session is created with org-scoped credentials and destroyed after completion.

📊 Data Access

All database queries pass through Postgres Row-Level Security. Even if application code has a bug, the DB layer enforces that Org A can never see Org B's goals, KPIs, check-ins, or conversation history.

💬 Slack Tokens

Each workspace has its own bot token (xoxb-...). Tokens are encrypted at rest, loaded per-request based on the incoming workspace ID, and never shared across orgs.

02 Tenant Lifecycle

1
Workspace Install

Admin clicks "Add to Slack" → OAuth consent → we receive access_token + team_id. A new org row is created with a UUID, the bot token is encrypted and stored, and the workspace metadata (name, icon, domain) is cached.

2
User Mapping

On first interaction (or via bulk sync), Slack user IDs are mapped to Brev user profiles. The mapping table (org_users) links slack_user_id → brev_user_id within the org scope. This enables the Operator to mention the right people and attribute check-ins correctly.

3
Agent Provisioning

Default agent configurations are created for the org: planning cadence (weekly), check-in schedule (Mon/Thu), escalation rules (48h no response → manager ping). These are stored in org_settings and can be customized via Slack commands or a future web dashboard.

4
Active Operation

The org is now live. Cron timers are registered, event handlers are active, and agents can be invoked. The org enters a steady state of: receive events → run agents → send messages → update state → repeat.

03 Data Isolation — Postgres RLS

Row-Level Security is the hard floor of tenant isolation. Even if application code is buggy, the database enforces boundaries.

-- Every tenant-scoped table has this policy:
ALTER TABLE goals ENABLE ROW LEVEL SECURITY;

CREATE POLICY org_isolation ON goals
  USING (org_id = current_setting('app.current_org_id')::uuid);

-- At the start of every request:
SET LOCAL app.current_org_id = 'org-uuid-here';

-- Tables with RLS enabled:
-- goals, kpis, check_ins, agent_sessions, org_users,
-- org_settings, conversations, slack_tokens, cron_schedules
Application Layer

Every DB connection sets app.current_org_id at transaction start. This is set by middleware, not by individual queries. Forgetting a WHERE clause is harmless — RLS blocks it.

API Layer

All API calls include org_id from the resolved Slack workspace. Internal service-to-service calls pass org_id as a header. There is no "global" query mode for tenant data.

04 Tenant Configuration

Per-org settings live in the org_settings table (JSONB column for flexibility, typed columns for critical paths).

Setting Storage Loaded
planning_cadenceTyped column (enum)At cron registration
checkin_scheduleTyped column (cron expr)At cron registration
escalation_rulesJSONBAt agent invocation
notification_channelsJSONB (channel IDs)At message send
timezoneTyped column (string)At schedule evaluation
custom_promptsJSONBAt agent invocation
feature_flagsJSONBCached in Redis (5m TTL)

Settings are loaded lazily — fetched when the relevant subsystem needs them, then cached in Redis with a 5-minute TTL keyed by org:{org_id}:settings. Cache is invalidated on write.

All infrastructure is defined in SST v3 (Ion). No manual AWS resources.

Compute
  • • Event Handlers: Lambda (128MB, 30s timeout) — receives Slack events, validates, enqueues
  • • Agent Workers: Lambda (512MB–1GB, 5min timeout) or ECS Fargate for long-running agents
  • • Cron Dispatcher: EventBridge Scheduler → Lambda that fans out per-org tasks to SQS
Data
  • • Postgres (RDS): db.t4g.medium initially, RLS on all tenant tables, pgBouncer for pooling
  • • Redis (ElastiCache): cache.t4g.micro — settings cache, rate limit counters, session state
  • • S3: Agent conversation logs (cold storage), exported reports

06 Agent Execution

Agent sessions are ephemeral, stateless, and org-scoped. Here's the lifecycle:

1. TRIGGER        Slack event / cron timer / manual command
       │
       ▼
2. ENQUEUE        SQS message: { org_id, agent_type, payload, trace_id }
       │
       ▼
3. DEQUEUE        Worker Lambda picks up message
       │
       ▼
4. INITIALIZE     - Set DB context: SET LOCAL app.current_org_id
                   - Load org settings from Redis/Postgres
                   - Load Slack bot token for the org
                   - Create agent_session row (status: running)
       │
       ▼
5. EXECUTE        - Fetch data from Brev API (goals, KPIs, users)
                   - Run LLM reasoning chain
                   - Determine actions (send messages, update data)
       │
       ▼
6. ACT            - Send Slack messages via org's bot token
                   - Write to Postgres (check-ins, status updates)
                   - Enqueue follow-up tasks if needed
       │
       ▼
7. CLEANUP        - Update agent_session row (status: complete, duration, tokens_used)
                   - Archive conversation log to S3
                   - Release DB connection back to pool

Key design decision: Agents are stateless between invocations. All state lives in Postgres and Redis. If a Lambda is recycled mid-execution, the SQS visibility timeout expires and the message is retried. Agent actions are idempotent (dedup key on agent_session + action hash).

07 Cron / Per-Tenant Scheduling

There is no global cron. Each org has its own schedules, evaluated in its own timezone.

EventBridge Scheduler (runs every minute)
       │
       ▼
Dispatcher Lambda
       │
       ├── SELECT * FROM cron_schedules
       │   WHERE next_run_at <= NOW()
       │   ORDER BY next_run_at
       │   LIMIT 100
       │   FOR UPDATE SKIP LOCKED   ← prevents double-dispatch
       │
       ├── For each schedule:
       │   ├── Enqueue SQS message { org_id, agent_type, schedule_id }
       │   └── UPDATE cron_schedules SET next_run_at = calculate_next()
       │
       └── Done (< 1 second typically)

Per-org schedules (examples):
  Acme Corp (America/New_York):  Mon 9:00am  → check-in nudges
  Acme Corp (America/New_York):  Fri 4:00pm  → weekly planning review
  Startup Inc (America/Los_Angeles): Tue 10:00am → check-in nudges
  Startup Inc (America/Los_Angeles): Thu 3:00pm  → mid-week pulse

The cron_schedules table stores: org_id, agent_type, cron_expr, timezone, next_run_at (UTC), last_run_at, enabled. The next_run_at is always stored in UTC but calculated from the org's timezone.

Cross-Service Events (SNS):
  ┌──────────────────────┐
  │ operator-events (SNS) │
  │                       │
  │ Topics:               │
  │  • org.installed      │──▶ Provision defaults, notify team
  │  • org.uninstalled    │──▶ Cleanup, archive data
  │  • agent.completed    │──▶ Metrics, billing counters
  │  • agent.failed       │──▶ Alerting, auto-retry logic
  │  • checkin.received   │──▶ Update goal status, notify manager
  └──────────────────────┘
Why FIFO + Message Groups?

Message group ID = org_id. This ensures per-org ordering (no two agents for the same org run concurrently) while allowing cross-org parallelism. Prevents race conditions on org state.

DLQ Strategy

Messages that fail 3 times land in the DLQ. A CloudWatch alarm fires if DLQ depth > 0. On-call reviews and either replays or discards. DLQ messages include full context for debugging.

09 Per-Tenant Cost Model

Costs are driven by three things: LLM tokens (dominant), compute (Lambda duration), and storage (DB rows). Infrastructure costs are shared.

Component Cost Driver Per-Org Attribution
LLM (Claude/GPT)Tokens consumedTracked per agent_session → billed to org
Lambda ComputeGB-secondsLogged per invocation with org_id tag
Postgres (RDS)Shared instanceRow counts per org for capacity planning
RedisShared instanceKey-prefix monitoring
SQSMessages sentNegligible ($0.40/million)
S3 (logs)Storage volumePrefix per org_id

Predictability levers: Per-org rate limits on agent invocations (max 50/day), token budgets per agent session (max 8K output tokens), and circuit breakers on runaway agents. Monthly cost cap per org tier.

Slack App Architecture

10 OAuth Install Flow (Multi-Tenant) 

Admin clicks "Add to Slack"
       │
       ▼
Slack OAuth Authorize URL
  https://slack.com/oauth/v2/authorize?
    client_id=...&
    scope=chat:write,commands,app_mentions:read,
          users:read,channels:read,groups:read&
    redirect_uri=https://api.brevoperator.com/slack/oauth/callback
       │
       ▼
Callback Handler (Lambda)
  1. Exchange code for token → { access_token, team: { id, name } }
  2. Check if org exists for team.id
     ├── Yes → update token, mark re-installed
     └── No  → CREATE org, CREATE org_settings (defaults)
  3. Encrypt token with KMS → store in slack_tokens table
  4. Publish SNS: org.installed { org_id, team_id }
  5. Send welcome DM to installing admin
  6. Redirect to success page

Scopes are minimal — we request only what's needed. No admin scopes. The bot token (xoxb-) is the only credential we store; we don't use user tokens.

11 Token Storage & Refresh

Storage
  • • Bot tokens encrypted with AWS KMS (AES-256-GCM)
  • • Stored in slack_tokens table (RLS-protected)
  • • Columns: org_id, encrypted_token, token_type, scopes, installed_at, installed_by
  • • Decrypted at request time, never cached in plaintext
Refresh Strategy
  • • Slack V2 OAuth bot tokens don't expire (no refresh needed)
  • • Token rotation happens on re-install (admin re-authorizes)
  • • If a token is revoked (app uninstalled), the tokens_revoked event marks the org as inactive
  • • Periodic health check: attempt auth.test per org, flag failures

12 Event Delivery — HTTP, Not Socket Mode

We use HTTP event subscriptions behind API Gateway, not Socket Mode. This is a deliberate choice for multi-tenant serverless:

✓ HTTP Mode (our choice)
  • • Serverless-compatible (Lambda)
  • • No persistent connections to maintain
  • • Auto-scales with API Gateway
  • • One endpoint for all workspaces
  • • Works behind CDN/WAF
✗ Socket Mode (rejected)
  • • Requires persistent WebSocket per workspace
  • • N workspaces = N connections = ECS/EC2 required
  • • Connection management complexity
  • • No auto-scaling benefit
  • • Reconnection handling burden
POST /slack/events  ←  Slack sends all workspace events here
       │
       ├── Verify request signature (HMAC-SHA256)
       ├── Parse event: extract team_id, event_type, payload
       ├── Respond 200 immediately (Slack requires < 3 seconds)
       │
       └── Async: enqueue to SQS for processing
           { org_id: resolve(team_id), event_type, payload }

13 Rate Limit Management

Slack rate limits are per-workspace, per-method. We track them per-org in Redis:

Redis keys (per org, per method):
  ratelimit:{org_id}:chat.postMessage  →  { remaining: 45, reset: 1710782400 }
  ratelimit:{org_id}:conversations.info →  { remaining: 18, reset: 1710782400 }

Rate limit middleware:
  1. Before API call: check Redis counter
  2. If remaining > 0: proceed, decrement
  3. If remaining = 0: queue for retry after reset time
  4. On 429 response: update Redis from Retry-After header
  5. Exponential backoff with jitter on retries

Slack Tier Limits (approximate):
  Tier 1 (chat.postMessage):  1 req/sec   (~50/min burst)
  Tier 2 (conversations.*):   20 req/min
  Tier 3 (users.info):        50 req/min
  Tier 4 (auth.test):         100+ req/min

Because agents send messages on behalf of different orgs, rate limits don't collide across tenants. Org A's message volume doesn't affect Org B. Each org's bot token has its own Slack rate limit bucket.

Part IV

Brev API Integration

14 Service-to-Service Auth

The Operator authenticates to the Brev API as a service principal, not as individual users. This is internal infrastructure, not a public API integration.

Authentication Flow:

  Operator Service                          Brev API
       │                                       │
       │  POST /api/v1/goals?org_id=...        │
       │  Authorization: Bearer {service_jwt}   │
       │  X-Operator-Org: {org_id}              │
       │──────────────────────────────────────▶│
       │                                       │
       │  Brev API validates:                  │
       │  1. JWT signature (shared secret/RSA) │
       │  2. Service ID in allowed list        │
       │  3. org_id matches JWT claims         │
       │                                       │
       │◀──────────────────────────────────────│
       │  200 OK { goals: [...] }              │

Service JWT Claims:
  {
    "sub": "operator-service",
    "iss": "brev-internal",
    "org_id": "uuid",       ← scoped to one org per request
    "permissions": ["goals:read", "goals:write", "kpis:read",
                    "users:read", "checkins:write"],
    "exp": 1710786000       ← short-lived (15 min)
  }

Key principle: The Operator never holds user credentials. It acts on behalf of an org, not a user. User identity is resolved from Slack user IDs mapped to Brev user IDs, but the API calls use the service token. This simplifies auth and avoids token-per-user sprawl.

15 Caching Strategy

✓ Cache (stable data)
  • User profiles — name, email, role, Slack ID mapping. TTL: 1 hour. Invalidate on user.updated webhook.
  • Goal trees — objective hierarchy, ownership. TTL: 15 min. Invalidate on goal.updated.
  • Team structure — who reports to whom. TTL: 1 hour.
  • Org settings — cadence, timezone, feature flags. TTL: 5 min.
✗ Always Fetch Fresh
  • KPI values — current metrics, progress percentages. Stale data = wrong decisions.
  • Check-in status — who has/hasn't checked in. Must be real-time accurate for nudges.
  • Recent activity — latest comments, updates, blockers. Context must be current.
  • Goal status — on-track/at-risk/off-track. Changes frequently.
Cache key pattern:
  brev:{org_id}:users           → hash of all user profiles
  brev:{org_id}:goals:tree      → serialized goal hierarchy
  brev:{org_id}:teams           → team structure
  org:{org_id}:settings         → org configuration

Invalidation:
  • TTL-based expiry (primary)
  • Webhook-driven invalidation (secondary, when available)
  • Manual flush via admin command: /operator cache-clear

Part V

Scaling

16 Horizontal Scaling

The entire compute layer is stateless. No sticky sessions, no in-memory state, no local disk dependencies.

Event Handlers

Lambda auto-scales to 1000 concurrent. Each invocation is independent. No warm-up needed for event ingestion (lightweight, <100ms).

Agent Workers

SQS concurrency controls how many agents run simultaneously. Start at 10, scale to 50+ as needed. Each worker is a Lambda that processes one SQS message.

Cron Dispatcher

Single Lambda, runs every minute, fans out to SQS. If org count grows past 10K, shard the dispatcher by org_id range. Not a concern until then.

18 Connection Pooling — Postgres

Lambda + Postgres is a known pain point. Without pooling, N concurrent Lambdas = N connections, easily exceeding Postgres limits.

Without pooling:
  50 Lambdas × 1 connection each = 50 connections
  Spike to 200 concurrent = 💥 connection limit exceeded

With RDS Proxy (our approach):
  200 Lambdas → RDS Proxy (connection pool) → Postgres
                    │
                    ├── Max 50 DB connections maintained
                    ├── Multiplexes Lambda connections
                    ├── Connection pinning for SET LOCAL (RLS)
                    └── Automatic failover on DB restart

Configuration:
  RDS Proxy max connections:  50  (80% of db.t4g.medium limit)
  Lambda reserved concurrency: 60  (agent workers)
  Idle timeout:               5 minutes
  Connection borrow timeout:  30 seconds

RLS + RDS Proxy gotcha: SET LOCAL requires transaction-level pinning. RDS Proxy supports this but it reduces multiplexing efficiency. We mitigate by keeping transactions short and using SET LOCAL (transaction-scoped, not session-scoped).
