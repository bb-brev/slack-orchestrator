L1 Proactive Planning

Cron-driven · Runs every 5 minutes · Scans organizational state · Initiates actions

1.1 State Scanning Algorithm

Every 5 minutes, the scanner runs a prioritized sweep across the entire org data model. Scans are ordered by urgency — the highest-priority items get evaluated first, and the scanner can short-circuit if the action budget (max actions per cycle) is exhausted.

Scan Priority Order

Priority Data Point Check Action Budget
P0 Blocked tasks Tasks marked blocked > 24h with no activity 3 actions/cycle
P0 Missed deadlines Due date passed, status ≠ completed 5 actions/cycle
P1 Stale commitments Action items from meetings with no update in >3 days 5 actions/cycle
P1 Goal health KR progress vs. expected trajectory (linear interpolation) 3 actions/cycle
P2 Metric anomalies Connected metrics deviating >2σ from 30-day trend 2 actions/cycle
P2 Upcoming deadlines Tasks due within 48h with <50% estimated completion 3 actions/cycle
P3 Check-in schedule People who haven't updated status in >5 business days 10 actions/cycle
P3 Reporting cadence Weekly/monthly reports due within next business day 2 actions/cycle

Scan Algorithm Pseudocode

async function runProactiveScan(org: Organization) {
  const budget = new ActionBudget(MAX_ACTIONS_PER_CYCLE)  // default: 20
  const now = DateTime.now()
  const bizHours = getBusinessHours(org.timezone)
  
  if (!bizHours.isActive(now)) return  // Skip outside business hours

  // P0: Critical — blocked tasks
  const blocked = await db.tasks.find({
    orgId: org.id,
    status: 'blocked',
    lastActivityAt: { $lt: now.minus({ hours: 24 }) }
  })
  for (const task of blocked) {
    if (!budget.canSpend()) break
    const action = buildEscalation(task, task.owner, task.owner.manager)
    await actionQueue.enqueue(action, { priority: 'critical', dedupKey: `blocked:${task.id}` })
    budget.spend()
  }

  // P0: Critical — missed deadlines
  const overdue = await db.tasks.find({
    orgId: org.id,
    dueDate: { $lt: now },
    status: { $nin: ['completed', 'cancelled'] }
  })
  for (const task of overdue) {
    if (!budget.canSpend()) break
    const daysPast = now.diff(task.dueDate, 'days')
    const action = daysPast > 3
      ? buildEscalation(task, task.owner, task.owner.manager)
      : buildFollowUp(task, task.owner)
    await actionQueue.enqueue(action, { priority: 'critical', dedupKey: `overdue:${task.id}` })
    budget.spend()
  }

  // P1: High — stale commitments from meetings
  const staleCommitments = await db.actionItems.find({
    orgId: org.id,
    source: 'meeting_transcript',
    status: 'open',
    lastUpdateAt: { $lt: now.minus({ days: 3 }) }
  })
  for (const item of staleCommitments) {
    if (!budget.canSpend()) break
    const action = buildContextualFollowUp(item, item.owner, {
      meetingName: item.sourceMeeting.name,
      transcriptMoment: item.sourceTimestamp,
      originalQuote: item.sourceQuote
    })
    await actionQueue.enqueue(action, { priority: 'high', dedupKey: `stale:${item.id}` })
    budget.spend()
  }

  // P1: High — goal health (KR trajectory check)
  const goals = await db.goals.find({ orgId: org.id, status: 'active' })
  for (const goal of goals) {
    if (!budget.canSpend()) break
    for (const kr of goal.keyResults) {
      const expected = linearInterpolation(kr.startValue, kr.targetValue, kr.startDate, kr.dueDate, now)
      const deviation = (expected - kr.currentValue) / kr.targetValue
      if (deviation > 0.15) {  // >15% behind trajectory
        const action = buildGoalAlert(goal, kr, kr.owner, deviation)
        await actionQueue.enqueue(action, { priority: 'high', dedupKey: `goal:${kr.id}` })
        budget.spend()
      }
    }
  }

  // P2–P3 continue similarly...
}

Action Types

Draft Check-in

Friendly DM asking for a status update. Used when there's no urgency but the person hasn't reported progress. Tone: casual, helpful.

Follow-up

Direct message with specific context about what needs updating. References the original commitment or deadline. Tone: professional, specific.

Escalation

Loops in the owner's manager. Only triggered after a previous follow-up went unanswered for 48h, or for P0 blockers. Tone: factual, no blame.

Report

Summary report posted to a channel or DM'd to a lead. Aggregates multiple data points into a structured update. Tone: analytical, data-driven.

1.3 Timing Strategy

The proactive loop respects human schedules. Actions are generated at scan time but delivered based on recipient context.

Delivery Windows

Timeline (recipient's local time):
                                                                              
  00:00    06:00    09:00    12:00    15:00    18:00    21:00    00:00
  ├────────┼────────┼────────┼────────┼────────┼────────┼────────┤
  │ SILENT │ SILENT │◀══ DELIVERY WINDOW ══▶│  QUIET  │ SILENT │
  │        │        │  Check-ins: 09–10     │  Only   │        │
  │        │        │  Follow-ups: 10–16    │  P0     │        │
  │        │        │  Reports: 08–09, 17   │  escal. │        │
  │        │        │  Escalations: 10–17   │         │        │

Scan runs globally every 5 min
  └─→ Actions enqueued with deliverAt timestamp per recipient TZ
  └─→ P0 escalations bypass quiet hours (configurable per org)
  └─→ Meeting buffer: no delivery 15min before/after calendar events

Multi-Timezone Handling

Each person in Brev has a timezone field (from Slack profile or manual override). The scanner produces actions immediately but the delivery layer holds them until the recipient's delivery window opens. For cross-timezone escalations (e.g., IC in PST, manager in CET), the action is delivered to the manager's timezone window since they're the decision-maker.

L2 Reactive Handling

Event-driven · Slack @mentions, reactions, thread replies · Instant response

2.1 @Mention Intent Detection

When a user @mentions Brev, the message is parsed through a two-stage intent detection pipeline: fast pattern matching for common operations, then LLM fallback for ambiguous requests.

Stage 1: Pattern Matching (< 50ms)

Pattern Intent CRUD Op Example
assign {thing} to {person} Assignment UPDATE "@brev assign the NPS goal to Sarah"
create {type} {details} Creation CREATE "@brev create a task to update the docs"
status of {thing} Query READ "@brev status of Q1 revenue goal"
update {thing} to {value} Update UPDATE "@brev update NPS to 47"
cancel/remove {thing} Deletion DELETE "@brev cancel the migration task"
summarize/report {scope} Report READ "@brev summarize this week's progress"

Stage 2: LLM Intent Resolution (< 2s)

For messages that don't match Stage 1 patterns, the full message + thread context is sent to the LLM with a structured output schema:

// LLM system prompt (abbreviated)
{
  role: "system",
  content: `You are the Brev Operator intent parser. Given a Slack message
  mentioning @brev, extract the user's intent as a structured operation.
  
  Output JSON:
  {
    "intent": "create|read|update|delete|assign|report|clarify",
    "entity_type": "task|goal|key_result|metric|cadence|person",
    "entity_query": "natural language description to fuzzy-match",
    "parameters": { ... },  // operation-specific
    "confidence": 0.0-1.0,
    "ambiguity_note": "optional explanation if confidence < 0.8"
  }
  
  If confidence < 0.6, set intent to "clarify" and include a suggested
  clarification question in ambiguity_note.`
}

// Entity resolution pipeline after LLM:
//   1. entity_query → fuzzy search against org's entities (trigram + embedding)
//   2. Top 3 matches returned, best match selected if confidence > 0.85
//   3. If multiple close matches → ask user to disambiguate
//   4. Person references resolved via Slack user ID + Brev person DB

2.2 Reaction-Based Approvals

When the Operator proposes an action (from any loop), it posts a message with a confirmation prompt. Users respond with emoji reactions to control execution.

✅ Execute

Approve and execute the proposed action immediately. The Operator performs the CRUD operation and confirms completion in-thread.

Who can: action owner, their manager, org admin

❌ Dismiss

Reject the proposed action. The Operator marks it as dismissed and won't re-propose for this item until state changes. Logged for learning.

Who can: anyone in the thread

👀 Acknowledge

Acknowledge without acting. The Operator records the acknowledgment and delays re-prompting by 48 hours. Useful for "I see this, will handle it myself."

Who can: action owner, their manager

🔄 Retry / Revise

Re-run the action with fresh context. Useful when the underlying data has changed since the proposal was made. Operator re-fetches and re-proposes.

Who can: anyone in the thread

Reaction Processing Pipeline

Slack Event: reaction_added
  │
  ├─ Is it on a Brev Operator message?  (check message metadata)
  │   └─ No → ignore
  │
  ├─ Extract: emoji, reactor_user_id, message_ts, channel
  │
  ├─ Authorization check:
  │   └─ Does reactor have permission for this reaction type?
  │       └─ No → add ⚠️ reaction, reply in thread: "Only {owner} can approve this"
  │
  ├─ Map emoji → action:
  │   ├─ ✅ → execute_proposed_action(message.metadata.action_id)
  │   ├─ ❌ → dismiss_action(action_id, reason: 'user_rejected')
  │   ├─ 👀 → acknowledge_action(action_id, snooze: 48h)
  │   └─ 🔄 → retry_action(action_id, refresh_context: true)
  │
  ├─ Execute and confirm:
  │   └─ Reply in thread with result or error
  │
  └─ Update action log with outcome + reactor identity

2.3 Thread Reply Routing

When a user replies in a thread that the Operator is participating in, the reply is routed back to the correct context — whether it's answering a check-in, providing a status update, or responding to a clarification request.

Thread Reply Routing:

  User replies in Brev thread
    │
    ├─ Lookup thread_ts → find originating action
    │   │
    │   ├─ Action type: CHECK_IN
    │   │   └─ Parse reply as status update
    │   │       ├─ Extract: progress %, blockers, timeline
    │   │       ├─ Update task/KR in Brev data model
    │   │       └─ Reply: "Got it! Updated {task} to {status}. ✅"
    │   │
    │   ├─ Action type: FOLLOW_UP
    │   │   └─ Parse reply as response to follow-up
    │   │       ├─ If contains update → apply update, clear follow-up timer
    │   │       ├─ If asks for help → route to manager or relevant person
    │   │       └─ If defers → set snooze, schedule re-follow-up
    │   │
    │   ├─ Action type: CLARIFICATION
    │   │   └─ Feed reply back into intent parser (Stage 2)
    │   │       ├─ If now confident → execute resolved intent
    │   │       └─ If still ambiguous → ask more specific question
    │   │
    │   ├─ Action type: PROPOSAL
    │   │   └─ Parse reply as approval/modification
    │   │       ├─ "yes" / "do it" / "approved" → execute
    │   │       ├─ "no" / "skip" / "not now" → dismiss
    │   │       └─ Other → treat as modification, re-propose
    │   │
    │   └─ Action type: REPORT
    │       └─ Parse reply as follow-up question
    │           └─ Generate additional analysis and reply in thread
    │
    └─ No originating action found
        └─ Treat as new @mention (enter intent detection pipeline)

2.4 Error Handling

When the Operator misunderstands intent or can't execute an action, it fails gracefully with clear communication and easy recovery paths.

Entity Not Found

👤 User:

"@brev assign the onboarding task to Mike"

🤖 Brev:

"I found 3 tasks with 'onboarding' in the name. Which one did you mean?
1️⃣ Onboarding flow redesign (Design team, due Mar 28)
2️⃣ Update onboarding docs (Eng team, due Apr 2)
3️⃣ New hire onboarding checklist (HR team, no due date)

Reply with the number, or say 'none' if I'm off."

Ambiguous Intent (confidence < 0.6)

👤 User:

"@brev deal with the revenue thing"

🤖 Brev:

"I want to help but I'm not sure what you need. Did you mean:
• 📊 Check status of the Q1 Revenue goal (currently 72% → target 85%)?
• 📝 Update the MRR metric?
• 👤 Assign someone to a revenue-related task?

Or just tell me more specifically what you need."

Permission Denied

If a user tries to modify an entity they don't own and aren't a manager/admin for, the Operator explains the restriction and offers to route the request to the appropriate person.

Execution Failure

If the CRUD operation fails (API error, validation error, conflict), the Operator explains what happened, what it tried, and suggests a retry or manual action. All failures are logged with full context for debugging.

Meeting-driven · Calendar-synced · Transcript-integrated · Continuous commitment tracking

3.1 Pre-Meeting: Agenda Generation

30 minutes before a scheduled cadence meeting, the Operator generates a data-driven agenda and posts it to the meeting's Slack channel.

Agenda Generation Algorithm

function generateAgenda(cadence: Cadence, meeting: Meeting): Agenda {
  const sections: AgendaSection[] = []
  const attendees = meeting.attendees
  const lastMeeting = cadence.previousMeeting
  const timeSince = now.diff(lastMeeting.endTime)

  // Section 1: CARRY-OVER ITEMS (highest priority)
  // Items from last meeting that are still open
  const carryOver = await db.actionItems.find({
    cadenceId: cadence.id,
    status: { $in: ['open', 'in_progress', 'blocked'] },
    createdBefore: lastMeeting.endTime
  })
  if (carryOver.length > 0) {
    sections.push({
      title: '🔄 Carry-Over Items',
      priority: 1,
      items: carryOver.map(item => ({
        text: item.title,
        owner: item.owner.name,
        status: item.status,
        daysSinceCreated: now.diff(item.createdAt, 'days'),
        context: item.lastUpdate || 'No updates since last meeting'
      }))
    })
  }

  // Section 2: GOAL & METRIC UPDATES
  // Pull latest data for goals/KRs relevant to this cadence's attendees
  const relevantGoals = await db.goals.find({
    ownerId: { $in: attendees.map(a => a.id) },
    status: 'active'
  })
  const goalUpdates = relevantGoals.map(goal => ({
    title: goal.title,
    progress: goal.overallProgress,
    trend: calculateTrend(goal, timeSince),  // improving/declining/flat
    keyHighlight: getMostNoteworthyKR(goal)  // biggest change since last meeting
  }))
  sections.push({
    title: '📊 Goal & Metric Updates',
    priority: 2,
    items: goalUpdates.sort((a, b) => urgencyScore(b) - urgencyScore(a))
  })

  // Section 3: RISKS & BLOCKERS
  // Anything flagged by the proactive loop since last meeting
  const risks = await db.actions.find({
    type: { $in: ['escalation', 'blocker_alert'] },
    createdAfter: lastMeeting.endTime,
    relatedTo: { $in: attendees.map(a => a.teamId) }
  })
  if (risks.length > 0) {
    sections.push({
      title: '⚠️ Risks & Blockers',
      priority: 3,
      items: risks
    })
  }

  // Section 4: NEW BUSINESS
  // Topics suggested by team members (via @brev add to agenda)
  const suggestions = await db.agendaItems.find({
    cadenceId: cadence.id,
    meetingId: meeting.id,
    source: 'user_suggestion'
  })
  sections.push({
    title: '💡 New Business',
    priority: 4,
    items: suggestions
  })

  // Section 5: TIME-BOXED by attendee count
  // Auto-allocate time per section based on meeting duration
  const totalMinutes = meeting.durationMinutes
  return allocateTime(sections, totalMinutes, attendees.length)
}

X.1 Loop Interaction Patterns

The loops are not isolated — they constantly feed into each other. Here are the primary interaction patterns:

Cross-Loop Data Flow:

  │                                                             │
  │  L3 (Cadence) ──creates──▶ Action Items ──tracked by──▶ L1 │
  │       │                                                  │  │

  │       └──agenda refs──▶   │ Shared State │  ◀──scans─────┘  │
  │                           │    Store     │                  │
  │  L1 (Proactive) ──────▶  │              │  ◀──writes── L2  │
  │       │                   └──────────────┘               │  │

  │       └──sends follow-up──▶ Slack DM ──reply──▶ L2 handles  │
  │                                                             │

Interaction Examples:

  1. L3 → L1: Meeting produces commitment → L1 proactive loop tracks it
     "Jake will get API keys by Friday" → L1 monitors, follows up Thu/Fri

  2. L1 → L2: Proactive follow-up → user replies → reactive handler
     L1 sends check-in DM → Jake replies "blocked on procurement" →
     L2 parses reply, updates task status to 'blocked', triggers L1 escalation

     L3 posts commitments → team member reacts ✅ → L2 processes approval

X.2 Task Lifecycle State Machine

Every task/action item in the system follows a unified state machine. All three loops read and write to these states.

                              ┌───────────┐
                              │ PROPOSED  │ ◀── L3 extracts from meeting
                              │           │ ◀── L2 creates from @mention
                              └─────┬─────┘ ◀── L1 suggests assignment
                                    │
                          ✅ react  │  ❌ react
                          or "yes"  │  or "no"
                                    │
                     ┌──────────────┼──────────────┐
                     ▼              │              ▼
              ┌───────────┐        │       ┌────────────┐
              │ APPROVED  │        │       │ DISMISSED  │ (terminal)
              └─────┬─────┘        │       └────────────┘
                    │              │
                    │ auto         │
                    ▼              │
              ┌───────────┐       │
              │  RUNNING  │ ◀─────┘ (auto-approve for high-confidence items)
              └──┬────┬───┘
                 │    │
      blocker    │    │ completed
      reported   │    │
                 ▼    ▼
          ┌──────────┐  ┌───────────┐
          │ BLOCKED  │  │ COMPLETED │ (terminal)
          └────┬─────┘  └───────────┘
               │
               │ unblocked (L2 reply or manual)
               ▼
          ┌───────────┐     ┌──────────┐
          │  RUNNING  │────▶│  FAILED  │ (terminal, can retry → RUNNING)
          └───────────┘     └──────────┘

State Transition Rules:
  • PROPOSED → APPROVED:  Only via human approval (✅ react or text)
  • PROPOSED → RUNNING:   Auto-approve if confidence > 0.95 AND org allows
  • APPROVED → RUNNING:   Automatic (no delay)
  • RUNNING → BLOCKED:    Owner reports blocker (via DM reply or @mention)
  • RUNNING → COMPLETED:  Owner confirms done, or metric target reached
  • RUNNING → FAILED:     Deadline passed + escalation exhausted
  • BLOCKED → RUNNING:    Blocker resolved (manual or via L2 update)
  • Any → DISMISSED:      ❌ reaction at any non-terminal state
  
Lock semantics:
  • Each transition acquires an optimistic lock on the task
  • If two loops try to transition simultaneously, last-write-wins
  • All transitions are logged in the action_history table

X.3 Conflict Resolution

When multiple loops attempt to act on the same item, the system uses these conflict resolution strategies:

Conflict: Duplicate Actions

Scenario: L1 proactive scan generates a follow-up for a stale task at the same time L3 is generating a pre-meeting agenda that includes the same stale task.

Resolution: Action deduplication layer. Every action has a dedupKey derived from (entity_id + action_type + time_window). If two actions have the same dedupKey within 4 hours, the higher-priority one wins and the other is suppressed. Suppressed actions are logged but not delivered.

Conflict: Race Condition on State

Scenario: L2 processes a user's reply ("it's done") and starts transitioning task to COMPLETED, while L1's scan (running concurrently) sees it as still RUNNING and enqueues a follow-up.

Resolution: Optimistic locking with version numbers. Each task has a version field. Transitions check version before writing. If L2 completes first (version 5 → 6), L1's follow-up action sees version mismatch at delivery time and is silently dropped. All state reads at delivery time, not enqueue time.

Conflict: Contradictory Actions

Scenario: L1 wants to escalate a blocked task to the manager. Meanwhile, the IC replied in the thread (L2) saying "unblocked, working on it now." L1 escalation is already in the queue.

Resolution: Delivery-time state validation. Before any action is delivered, the delivery layer re-checks the entity's current state. If the state no longer matches the action's preconditions (e.g., task is no longer blocked), the action is cancelled with reason precondition_stale.

Conflict: Message Flooding

Scenario: Multiple loops generate messages to the same person in a short window — creating an annoying burst of notifications.

Resolution: Per-person rate limiter. Max 3 Operator messages per person per hour (configurable per org). If limit is hit, lower-priority messages are batched into a single digest delivered at the next available slot. P0 escalations bypass the rate limit but still count toward the budget.

Priority Hierarchy (When Loops Conflict)

Priority (highest to lowest):

  1. L2 Reactive — User-initiated actions always win
     Rationale: Human just asked for something → do it now

  2. L3 Cadence — Meeting-driven actions take precedence
     Rationale: Team is actively in a meeting → context is fresh

  3. L1 Proactive — Background scanning is lowest priority
     Rationale: Can always wait for the next cycle

Exception: L1 P0 escalations (blocked > 24h, missed critical deadlines)
  → These jump to priority 1.5 (above cadence, below reactive)
